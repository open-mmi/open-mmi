# Open MMI vehicle-action registry

> Generated from `canbusd/data/vehicle-actions.v1.json`. Do not edit this file by hand.

Bindings translate canonical vehicle events into these stable local behaviors. The action
identifier describes what Open MMI should do; Python modules, function names, commands and
packages remain private implementation details owned by the registry.

> **The action registry is a continuity checkpoint, not a walled garden.** Reuse an
> existing human-readable behavior when it fits, or propose a genuinely new action,
> implementation, documentation and tests in the same pull request.

## Binding example

A maintained binding names a stable action, not its Python implementation:

```json
{
  "mute_toggle": {
    "action": "media.mute.toggle"
  },
  "brightness_level": {
    "action": "display.brightness.set"
  }
}
```

The event says what happened. The action says what Open MMI should do. The registry
privately resolves the action to the current implementation.

## Compatibility boundary

Existing custom bindings using `module`, `func` and `args` remain supported during the
migration window and receive a validation warning. Maintained bindings must use canonical
action identifiers. New contributions must not expose Python implementation names in the
maintained binding catalogue.

## Reuse or propose

Search using ordinary language before adding a new action:

```bash
open-mmi-config vehicle-setup actions --search "audio mute"
open-mmi-config vehicle-setup actions --search "screen power off"
```

Check an exact proposed identifier:

```bash
open-mmi-config vehicle-setup actions --check media.mute.toggle
open-mmi-config vehicle-setup actions --check sound_module_off
```

A new action proposal should define the universal behavior, argument contract, event-payload
contract, requirements, implementation and tests. No prior permission is required; registry
review prevents duplicate dialects and implementation details from becoming public API.

## Registry rules

- Existing stable action identifiers never silently change meaning.
- Action names describe local behavior, not the event that triggered it.
- Manufacturer names, CAN IDs, Python modules, functions and executable names do not belong in action IDs.
- Configured arguments and event payloads are separate, validated contracts.
- Deprecated aliases are migration diagnostics only; maintained bindings use the canonical identifier.
- Availability requirements are descriptive and may be surfaced by future capability tooling.
